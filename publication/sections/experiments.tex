\subsection{Distributional Similarity}

For numeric columns, planned metrics include univariate summary statistics,
histogram overlap, Wasserstein distance, and pairwise correlation differences.
For categorical columns, planned metrics include category coverage, frequency
divergence, and rare-category retention.

\subsection{Downstream Predictive Utility}

For supervised datasets, we plan to train predictive models on generated data
and evaluate them on held-out real data. This train-on-synthetic, test-on-real
protocol will provide a practical measure of whether generated samples preserve
useful predictive structure.

\subsection{Ablations}

Planned ablations include:

\begin{itemize}
  \item frequency encoding versus helper-column embedding for categorical
  columns;
  \item PCA, MDS, and nonlinear embedding choices;
  \item exact versus approximate nearest-neighbour backends;
  \item mutual-neighbour sampling with and without fallback edges;
  \item manually specified configuration versus LLM-assisted configuration;
  \item generation with and without sensitive-value replacement for selected
  identifier columns.
\end{itemize}

\subsection{Usability And Safeguard Checks}

The LLM-assisted configuration mode will be evaluated as a usability feature
rather than as an independent modelling component. Planned checks include the
rate at which returned configurations are syntactically valid, whether selected
helper columns are type-compatible, and whether user overrides are preserved.
For the anonymization layer, planned checks include exact source-value overlap
in selected columns before and after generation, consistency of repeated
surrogate mappings, and failure behaviour when the model proposes a replacement
that collides with an original value.

\subsection{Preliminary Baseline Comparisons}

The current experiment scaffold includes Adult Census Income and Titanic as
initial mixed-type datasets. For each dataset, we compare simple baselines
against three DataFrameSampler configurations: default settings, a manual
configuration, and a fixed LLM-style configuration recorded in the notebook for
reproducibility. The LLM-style condition should be read as an experiment proxy
for the kind of structured recommendation returned by the LLM interface; the
notebooks do not call an external API during execution.
