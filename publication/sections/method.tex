\section{Method}

\subsection{Column Vectorization}

Let $D$ be an input dataframe with $m$ rows and $d$ columns. Numeric columns are
converted to floating point values and missing values are imputed with the
column median. Before vectorization, columns with a clear human scale can be
encoded directly: booleans become $0/1$, binary labels such as female/male can
be encoded as $0/1$, ordinal labels such as passenger class can be encoded as
$1/2/3$, and derived ordered labels such as woman/child/man can be encoded as
$0/0.5/1$. Redundant aliases are removed at the same stage so that the latent
geometry does not overweight one fact. In the Titanic notebook, for example,
\texttt{pclass}, \texttt{embark\_town}, and \texttt{survived} are preserved,
while their aliases \texttt{class}, \texttt{embarked}, and \texttt{alive} are
dropped. For a remaining non-numeric column $c$, DataFrameSampler supports two
vectorization strategies.

If no helper columns are provided, each category is represented by its empirical
frequency. This is deliberately simple, but it only gives a weak ordering:
categories with similar counts become nearby even if they have no semantic
relationship. The more important case is helper-column vectorization. If helper
columns are provided, each row receives a helper vector
$h_i \in \mathbb{R}^{k}$ constructed from the selected numeric helper columns.
An embedding function
\[
  f: \mathbb{R}^{k} \rightarrow \mathbb{R}
\]
maps each helper vector to a scalar used to discretize column $c$. In the
implementation, $f$ may be selected from several scikit-learn reducers,
including PCA, MDS, Kernel PCA, Isomap, locally linear embedding, and related
methods. Users may also supply custom objects implementing a scikit-learn-like
\texttt{fit\_transform}, \texttt{fit}/\texttt{transform}, or \texttt{transform}
interface.

The purpose of this step is to turn a symbolic category into a one-dimensional
trace that preserves a useful notion of neighbourhood for that column. For
example, suppose a table contains a \texttt{nation} column and also contains
latitude and longitude columns describing the nation centroid. A user can choose
latitude and longitude as helper columns and use an embedding such as MDS to
place nations on a one-dimensional curve through the geographic helper space.
Nearby points on this curve then correspond, approximately, to nearby nations.
If a row containing Spain has a neighbour containing Italy, and Italy's
neighbour in the embedded trace is Austria, the local difference points
north-east in the geographic helper space. Applying an analogous local movement
from Spain may decode to a nearby country such as France, rather than to an
unrelated country chosen only because it has a similar empirical frequency.

This helper selection is therefore a semantic modelling decision. The user, or
the optional LLM-assisted configuration layer, must decide which numeric columns
express the context in which a categorical column should be ordered. For a job
title this might be seniority, compensation, and department size; for a medical
diagnosis it might be laboratory values and age; for a banking campaign outcome
it might be contact duration, prior contacts, and economic indicators. The
notebooks report these choices explicitly as a vectorization plan before any
sampling results are shown. Pairwise exploratory plots are drawn after this
numeric reduction step, using the same transformed coordinates used by the
sampler, so visual inspection refers to the operative latent geometry rather
than only to raw numeric columns.

\subsection{Discretization And Decoding}

After vectorization, each column is discretized into up to $b$ bins using a
one-dimensional binning procedure. Each row $x_i$ is encoded as an integer
latent vector
\[
  z_i \in \{0, \ldots, b-1\}^{d}.
\]
For every column and bin, the encoder stores the original observed values that
fell into that bin. Decoding a generated latent vector therefore consists of
sampling original values from the selected bins. If a generated bin is empty,
the implementation searches for the nearest non-empty bin in that column.

\subsection{Neighbour-Based Latent Sampling}

Let $Z = \{z_i\}_{i=1}^{m}$ be the encoded latent matrix. DataFrameSampler builds
a nearest-neighbour graph in this latent space. By default, only mutual
nearest-neighbour relationships are retained. If a row has no mutual neighbours,
a nearest-neighbour fallback is used to preserve sampleability.

The geometric intuition is that the encoded rows lie on an empirical manifold
inside the discretized latent space. Around any row, mutual nearest neighbours
provide a local, symmetric estimate of which observations occupy the same
small neighbourhood of that manifold. A difference vector between two nearby
neighbours is therefore treated as a first-order approximation to a direction in
the local tangent space. Sampling transfers such a local tangent direction to a
nearby anchor point, with the aim of moving along the empirical manifold rather
than jumping to an unrelated part of the latent space. This is a modelling
intuition and an inspectable construction, not a formal guarantee that generated
rows remain on a smooth manifold.

To generate a new latent row, the sampler selects an anchor row $z_a$, a
neighbour $z_b$ of that anchor, and a neighbour $z_c$ of $z_b$. It then forms
\[
  \tilde{z} = z_a + \alpha (z_c - z_b),
\]
where $\alpha$ is sampled from a configured interpolation range. The generated
latent vector is rounded to integer bin indexes, clipped to valid ranges, and
decoded back to the original dataframe space.

The operation is intentionally simple to inspect. For a generated example, the
user can identify the anchor row, the neighbour chain, the latent difference
that was transferred, and the bins used during decoding. This is an important
design choice for applied domains where generated examples may be shown to
business users, clinicians, auditors, or software teams that need a concrete
explanation rather than a black-box generation claim.

\subsection{Generation Trace}

For a generated row $\tilde{x}$, a trace is defined as the tuple
\[
  \tau(\tilde{x}) = (a, b, c, z_a, z_b, z_c, \tilde{z}, B),
\]
where $a$, $b$, and $c$ are the anchor and neighbour indexes, $z_a$, $z_b$, and
$z_c$ are their latent bin vectors, $\tilde{z}$ is the generated latent vector,
and $B$ records the per-column bins used for decoding. This trace is the object
that supports the paper's inspectability claim: it connects an output row to a
small number of source rows and a simple arithmetic operation.

The operational procedure is:
\begin{enumerate}
  \item optionally replace selected sensitive columns and verify no exact or
  normalized overlap with source values;
  \item vectorize the dataframe and encode each column into bins;
  \item build the nearest-neighbour graph, using fallback nearest neighbours
  when mutual-neighbour search leaves a row isolated;
  \item sample an anchor-neighbour chain and transfer a local latent difference;
  \item clip and decode the generated bin vector into a dataframe row;
  \item verify selected anonymized columns against the original source values
  before returning or saving generated rows.
\end{enumerate}

\subsection{LLM-Assisted Configuration}

Choosing helper columns for categorical vectorization can require semantic
knowledge of column roles. The package therefore includes an optional
OpenAI-assisted mode that treats configuration as a structured recommendation
task. The helper sends a compact dataframe profile to an LLM, including column
names, dtypes, missing counts, unique counts, examples, and basic numeric
summaries. The model returns a JSON-compatible configuration containing
candidate helper columns, sampled columns, an embedding method, and a
nearest-neighbour backend.

The LLM is not part of the generative model itself. It is used only to reduce
the friction of selecting a reasonable configuration. Explicit user choices
take precedence over model recommendations. In the command-line interface, the
\texttt{-A} flag asks the model to fill only those options that were not already
specified by the user. In the Python API, the same helper returns a dictionary
that can be inspected, edited, logged, or passed into the sampler.

\subsection{Sensitive-Value Replacement}

For workflows involving direct identifiers, the package can replace selected
sensitive columns before fitting the sampler. Let $S$ be the set of columns
marked for replacement. For each $c \in S$, the implementation extracts the
distinct non-null source values and asks an LLM to produce surrogate values of
the same broad type, such as plausible names, organizations, or labels. The
surrogate mapping is then applied to the dataframe before vectorization,
discretization, neighbour construction, or sampling occurs.

After replacement, the implementation checks that no generated surrogate value
has exact or normalized string overlap with the original values in the selected
columns. The sampler is then fit on the surrogate dataframe. Generated samples
are checked again against the original source values before being returned or
written to disk. This design ensures that the core sampler never needs to ingest
the original sensitive values for those columns. The procedure reduces exact
identifier re-emission in selected columns, but it is not a formal privacy
guarantee and does not provide differential privacy, $k$-anonymity, or
protection against linkage attacks.
